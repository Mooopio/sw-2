S
SRP
Single Responsibility Principle
Coined by Robert C. Martin (Uncle Bob)
🎯 Two Definitions — Both are EXAM-READY
Definition 1: "Every software component should have one and only one responsibility."
Definition 2 (Uncle Bob): "Every software component should have one and only one reason to change."

Component = class, method, or module. Quote: Heraclitus — "The only thing that is constant is change."

📦 Concept 1 — Cohesion

Cohesion = How RELATED are parts inside a component

High cohesion = parts are closely related → ✅ GOOD
Low cohesion = parts have nothing to do with each other → ❌ BAD

🗑️ Real-World Analogy

Unsegregated trash bin = LOW cohesion (everything mixed). Segregated bins (plastic / paper / organic) = HIGH cohesion. Each bin has ONE type of content.

❌ Bad — Low Cohesion (Square class)

class Square { // Measurements calculateArea() calculatePerimeter() // Rendering — NOT related! draw() rotate() }

calculatePerimeter() and draw() have NO relation → LOW cohesion

✅ Good — High Cohesion (2 classes)

class Square { // Measurements only calculateArea() calculatePerimeter() } class SquareUI { // Rendering only draw() rotate() }

Each class has ONE job → HIGH cohesion ✅

🧠 MEMORY TRICK

HIGH cohesion = HIGH grade. High is always good for cohesion. Ask yourself: "Do all methods in this class actually BELONG together?" If yes → high cohesion ✅

🔗 Concept 2 — Coupling

Coupling = Level of inter-dependency between components

Tight coupling = change one class → must change another too → ❌ BAD
Loose coupling = classes are independent → ✅ GOOD

🚂 Train Track Analogy

Train tracks have FIXED gauge width → a train CANNOT run on a different track → Tightly coupled. In software, tight coupling = bad. You want trains that can switch tracks (loose coupling).

❌ Tight Coupling — BAD

class Student { // Core data + DB logic mixed! save() { // MySQL code directly Class.forName("com.mysql.jdbc.Driver"); // Change DB = rewrite Student! } }

3 reasons to change: id format, name format, DB backend

✅ Loose Coupling — GOOD

class Student { save() { new StudentRepository().save(this); } } class StudentRepository { // ALL DB code here save(student) { ... } }

Change DB → only modify Repository, Student unchanged ✅

📊 Student Class — Reasons to Change
ClassReasons to ChangeOK?
Student (original)id format + name format + DB backend❌ 3 reasons
Student (fixed)id format + name format → "student profile"✅ 1 reason
StudentRepositoryDB backend only✅ 1 reason

If 2 reasons are CLOSELY RELATED, you can combine them into 1. "id format" + "name format" = "changes to student profile"

🧠 MEMORY TRICK

SRP Formula: HIGH Cohesion + LOOSE Coupling = SRP ✅
Ask: "How many reasons does this class have to change?" → Should be exactly ONE.

O
OCP
Open / Closed Principle
Open for Extension. Closed for Modification.
🎯 Core Definition
"Software entities should be open to extension but closed to modification."

OPEN FOR EXTENSION

Design classes so NEW functionality can be added as new requirements arrive → adapt to new needs

CLOSED FOR MODIFICATION

Once a class is developed, NEVER modify it (except to fix bugs) → stable, no side effects

⚽ How to Achieve OCP

Use interfaces / abstract classes instead of concrete classes for dependencies. Add new features by creating NEW classes that implement the interface. This avoids touching existing, working code.

⚽ Football Match Example — Strategy Pattern

Scenario: Al-Ahly (Attack +2) vs Zamalek (Defend +1)

❌ Violates OCP — if/else chains

Play() { if (Team1 is Ahly) Ahly is attacker else if (Team1 is Zamalk) Zamalk is defend } Welcome() { if (Team1 is Ahly) ... else if (Team1 is Zamalk) ... } SetScore(winner) { if winner is Ahly → +2 else if winner is Zamalk → +1 }

Adding a new team = modifying existing code. Violates OCP!

✅ Follows OCP — Strategy Pattern

abstract class Strategy { name, scorePoint play() setScorePoint() } class Attack extends Strategy { play() → Attack! setScorePoint() → +2 } class Defend extends Strategy { play() → Defend! setScorePoint() → +1 } Match.play() → Team1.play()

New team = new Strategy subclass. Zero modification to Match! ✅

🎨 Strategy Design Pattern — 3 Parts to Know

PartRoleIn Example
ContextHolds reference to a Strategy, talks via interface onlyMatch class / Team class
Strategy InterfaceCommon interface, declares methods context usesAbstract class Strategy (play, setScorePoint)
Concrete StrategiesImplement different variations of the algorithmAttack class, Defend class
🧠 MEMORY TRICK

"Add, don't edit." Want new behavior? Add a new class. Never touch the old one. OCP = Extension YES, Modification NO.

OCP links to DIP: one of the ways to achieve OCP is to use the Dependency Inversion Principle (abstractions!).

L
LSP
Liskov Substitution Principle
By Barbara Liskov — 1987 Conference Keynote
🎯 Core Definition
"If class B is a subclass of class A, then wherever A is expected, B can be used instead — without breaking the system."

→ Objects of a superclass should be replaceable with objects of its subclasses without breaking the system.
→ A subclass should NOT be more restrictive than the behavior specified by the superclass.
→ This principle ensures inheritance is used CORRECTLY.
→ Red flag: If an override method does nothing OR throws an exception → you're probably violating LSP!

📋 The 4 Conditions for LSP Compliance — MEMORIZE THESE
  • Condition 1 — Contravariance (Method Signatures must match):
    Method parameters must be the same OR MORE GENERIC than the parent's method parameters. You cannot narrow them. (Allowed in PHP, NOT in Java.)
  • Condition 2 — Preconditions cannot be STRENGTHENED in subtype:
    An inherited method cannot have MORE conditions than the parent's method. Example: Parent accepts any integer → child only accepts positive integers → ❌ VIOLATES LSP. You're making conditions STRICTER (stronger pre-conditions = bad).
  • Condition 3 — Postconditions cannot be WEAKENED in subtype:
    Postconditions must be AT LEAST EQUAL to the parent's. Example: Parent closes DB connection at end of method → child keeps it open → ❌ VIOLATES LSP. You're delivering LESS than promised (weaker post-conditions = bad).
  • Condition 4 — Exception types must match:
    If parent throws FileNotFoundException, the overriding method must also throw FileNotFoundException. Cannot throw a different or unexpected exception type.
🧠 TRICK to remember Pre vs Post conditions

PREconditions = conditions BEFORE the method runs (input constraints) → Cannot make stricter (more restrictive)
POSTconditions = what the method GUARANTEES after running (output/state) → Cannot weaken (deliver less)

🚗🚲 Classic Example — Vehicle / Bicycle

❌ Violates LSP

class Bicycle extends Vehicle { startEngine() { throw new Exception( "Not supported" ); // 💥 VIOLATES LSP } } // In main: for(Vehicle v : vehicleList) { v.startEngine(); // CRASH! 💥 }

✅ Follows LSP — Correct Hierarchy

class Vehicle { name, speed } class VehicleWithEngines extends Vehicle { startEngine() } class VehicleWithoutEngines extends Vehicle { startMoving() } class Car extends VehicleWithEngines class Bicycle extends VehicleWithoutEngines
⚠️ 3 Situations that Violate LSP
  • Subclass returns an object completely different from what the superclass returns
  • Subclass throws an exception NOT defined in the superclass
  • There are side effects in subclass methods that were NOT part of the superclass definition
☕ Java Note — Covariant Return Type (Java 5+)

Before Java 5: override → parameters AND return type must match exactly.
Java 5+: Covariant return type = overriding method can return a subclass of the return type declared in the superclass.
Example: Parent returns Circle → Child can return Square (if Square extends Circle). ONLY works if child return type is a subtype of parent return type.

class Circle {} class Square extends Circle {} class ParentClass { public Circle method1(Circle c) { return new Circle(); } } class ChildClass extends ParentClass { @Override public Square method1(Circle c) { return new Square(); } // ✅ OK! Square IS-A Circle }
🧠 MEMORY TRICK

LSP = "Can I replace parent with child and nothing breaks?"
If your child class has methods that throw exceptions or do nothing when called → RED ALERT → fix the inheritance hierarchy!

I
ISP
Interface Segregation Principle
By Robert C. Martin — Consulting for Xerox printer systems
🎯 Core Definition
"Clients should NOT be forced to depend upon interfaces that they do not use."

RULE 1

No class should be forced to implement method(s) of an interface they don't use

RULE 2

Instead of big "fat interfaces", create multiple smaller interfaces — clients only think about methods of interest to them

🔗 Connection to SRP

ISP is like SRP but for interfaces. SRP = each class/module has one purpose. ISP = each interface has one purpose. Same idea, different level!

🖨️ Printer Example

❌ FAT Interface — Violates ISP

interface PrinterTasks { Print(content) Scan(content) Fax(content) PrintDuplex(content) } // LiquidInkjetPrinter doesn't have Fax! class LiquidInkjetPrinter implements PrinterTasks { Fax() { throw new UnsupportedOperationException(); } PrintDuplex() { throw new UnsupportedOperationException(); } }

LiquidInkjet forced to implement Fax+Duplex it doesn't support → LSP violation too!

✅ Segregated Interfaces — Follows ISP

interface PrintingTasks { Print() } interface ScanTasks { Scan() } interface FaxTasks { Fax() } interface DuplexTasks { PrintDuplex() } // HP LaserJet needs everything class HPLaserJetPrinter implements PrintingTasks, ScanTasks, FaxTasks, DuplexTasks { } // Inkjet only needs Print + Scan class LiquidInkjetPrinter implements PrintingTasks, ScanTasks { }
🧠 MEMORY TRICK

"Don't force a simple printer to know about faxing."
Fat interface = forced methods you'll never use = bad design. Split it! Many small interfaces > one giant interface.
Key indicator: If a class implements an interface but throws UnsupportedOperationException → ISP violated!

D
DIP + DI
Dependency Inversion Principle
+ Dependency Injection (Implementation technique)
🎯 Core Definition — DIP
"High-level modules should NOT depend on low-level modules. Both should depend on abstractions. Abstractions should NOT depend on details. Details should depend upon abstractions."

↓ Normal (bad): High-level → depends on → Low-level
↑ Inverted (good): High-level → depends on → Abstraction ← depends on ← Low-level

Why? Code violating DIP = too tightly coupled = hard to maintain, hard to read, prone to side effects.

🛒 E-Commerce Store + Payment Gateway Example

❌ Violates DIP — Store depends on Stripe directly

class Store { Stripe stripe; // hard-coded! Store() { stripe = new Stripe(); // tightly coupled } sellItem() { stripe.processPayment(); } }

Stripe goes down? You MUST modify Store class. Multiple instances to change = bugs everywhere.

✅ Follows DIP — Depends on Abstraction

interface PaymentGateway { processPayment(); } class Stripe implements PaymentGateway {} class Paypal implements PaymentGateway {} class Store { PaymentGateway pg; // abstraction! setPaymentGateway(PaymentGateway pg){} sellItem() { pg.processPayment(); } }

Switch Stripe→Paypal? Zero changes to Store! Just inject a different implementation ✅

👔 Manager / Workers Example (DIP)

❌ Manager knows specific types

class Manager { developers = [] designers = [] testers = [] addDeveloper(dev) addDesigner(design) addTester(tester) }

Manager must know ALL types upfront. Add new worker type = modify Manager class!

✅ Manager depends on abstraction

interface Worker { work() } class Developer implements Worker {} class Designer implements Worker {} class Manager { workers = [] // just Worker list! addWorker(Worker w) {} }

New worker type? Just implement Worker. Manager has NO idea about specific types → truly decoupled ✅

💉 Dependency Injection (DI) — How to IMPLEMENT DIP
DI = being able to pass (inject) the dependencies when required instead of initializing them inside the recipient class.

DI resolves dependencies at runtime rather than compile time.
DI is a design pattern, not a framework. It's one way of implementing Inversion of Control (IoC).
IoC = inverting additional responsibilities of a class (not the main one).

❌ WITHOUT DI — Problems
  • Code is inflexible — change one class = must change the dependent class too, must recompile
  • Code is hard to unit test — testing one class forces testing all its dependencies
  • Code is hard to reuse — classes are tightly coupled

3 Types of Dependency Injection

🏗️

Constructor Injection

Dependency passed via constructor. Used when dependency must stay the same for the object's lifetime.

Store(PaymentGateway pg) { this.pg = pg; }
⚙️

Setter Injection

Dependent class has a public setter method through which the dependency is injected.

setPaymentGateway( PaymentGateway pg) { this.pg = pg; }
🔌

Interface Injection

An interface provides an injector method. Clients implement the interface and override the injector method.

interface Injector { setPaymentGateway(pg); } Store implements Injector
✅ Advantages of DI
  • Makes testing easier — inject mock dependencies
  • Reduces coupling between client and dependency classes
  • Code is easier to maintain, reuse and extend
🧠 MEMORY TRICK — DIP vs DI

DIP = the PRINCIPLE (what to do): depend on abstractions, not concretions.
DI = the TECHNIQUE (how to do it): inject dependencies instead of hardcoding them.
DI implements DIP. DIP enables OCP.

"The client does NOT control how objects are created — it works with an implementation through an INTERFACE." = this is also why it's called Inversion of Control (IoC).

⚡ Exam Flash Quiz — Click to Reveal

+
What is Cohesion? Is high or low cohesion better?
Cohesion = the degree to which the various parts of a software component are RELATED. HIGH cohesion = better. It means all methods in a class actually belong together and serve one purpose. High cohesion helps attain better adherence to SRP.
+
What is Coupling? Is tight or loose coupling better?
Coupling = level of inter-dependency between software components. How much does changing one thing force changes in another? LOOSE coupling = better. Tight coupling (like a train on one track) is undesirable in software because changes cascade everywhere.
+
Uncle Bob's exact definition of SRP?
"Every software component should have one and only one REASON TO CHANGE." (Not just one responsibility — one REASON TO CHANGE. Both definitions are valid, but Uncle Bob prefers this one.)
+
What are the 3 types of Dependency Injection?
1. Constructor Injection — pass dependency via constructor. 2. Setter Injection — public setter method to inject. 3. Interface Injection — implement an injector interface with a setter method.
+
Who defined LSP, when, and where?
Barbara Liskov defined it in a 1987 conference keynote. It ensures inheritance is used correctly. If B is a subclass of A, wherever A is expected, B can be used without breaking the system.
+
Who defined ISP, and why?
Robert C. Martin, while consulting for Xerox to build software for their new printer systems. He defined it as: "Clients should not be forced to depend upon interfaces that they do not use."
+
How does DIP connect to OCP?
One of the key ways to achieve OCP is to use DIP! By depending on abstractions (interfaces), you can add new behavior (OCP = open for extension) by creating new concrete classes WITHOUT modifying existing code (OCP = closed for modification).
+
What is Inversion of Control (IoC)?
IoC is basically used to invert different kinds of ADDITIONAL responsibilities of a class rather than the main responsibility. DI is one way of implementing IoC. The client doesn't control how objects are created — it works via an interface. This is inverse to trivial programming, hence the name "Inversion of Control".
+
LSP Condition 2 vs Condition 3 — what's the difference?
Condition 2 (Preconditions): Input constraints CANNOT be made STRICTER in subclass. If parent accepts any integer, child cannot restrict to positive only.

Condition 3 (Postconditions): What the method guarantees after running CANNOT be WEAKENED. If parent closes DB connection, child cannot keep it open. You must deliver at least what the parent promised.
+
What is Covariant Return Type in Java?
Introduced in Java 5. An overriding method can return a SUBTYPE of the return type declared in the superclass method with the same signature. Example: Parent returns Circle → Child can return Square (if Square extends Circle). You CANNOT return a completely unrelated type.
+
What are the 3 components of the Strategy Design Pattern?
1. Context — maintains reference to a concrete strategy, communicates ONLY via the strategy interface. 2. Strategy Interface — common to all concrete strategies, declares the method the context uses. 3. Concrete Strategies — implement different variations of the algorithm.
