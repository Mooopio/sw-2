Creational Patterns

About object creation — how and when to create objects flexibly

Factory Method
🏭
"Define an interface for creating an object, but let subclasses decide which class to instantiate."

The Problem

  • Logistics app starts with only Trucks. Later, Ships are needed.
  • Adding new transport types means changing the Logistics class every time → violates OCP!

The Solution – Components

«interface» Transport
+ deliver()
←implements
Truck
Ship
«abstract» Logistics
+ createTransport() ← FACTORY METHOD
+ planDelivery()
↓extends
RoadLogistics
→ returns Truck
SeaLogistics
→ returns Ship
❌ BEFORE (Bad)
if type=="Truck": new Truck()
elif type=="Ship": new Ship()
→ Must MODIFY to add Bus!
✅ AFTER (Good)
class BusLogistics(Logistics):
def createTransport():
return Bus()
→ NO modification needed!
🧠 Exam Trick: Factory Method = one product type, created by subclass override. Key method is always createXxx(). It solves OCP — open for extension, closed for modification.

Open/Closed Principle (OCP)

Software should be Open for Extension (add new classes) but Closed for Modification (don't touch existing code). Factory Method enforces this perfectly.

Real Examples from Slides

  • Dialog Buttons: WindowsDialog returns WindowsButton, WebDialog returns HTMLButton — same Dialog logic, different buttons per OS
  • Shape Area: Before: AreaCalculator checks instanceof. After: Each Shape has its own area() method
  • Logistics: RoadLogistics vs SeaLogistics
Builder
🏠
"Separate the construction of a complex object from its representation — build step by step."

The Problem

  • House needs: windows, doors, rooms, garage, pool, statues, garden...
  • Giant constructor: new House(4, 2, 4, true, null, null, null) → Ugly! What is each null?
  • Creating subclasses for every combo → Explosion of classes!

Method Chaining Pattern (Fluent Interface)

  • Each setter returns $this → enables chaining
  • Final method is always build() or getResult()
  • CarBuilder example: set_year().set_color().set_engine().build()

SQL Query Builder (Exam Example!)

  • MysqlQueryBuilder: .select().where().limit().getSQL()
  • PostgresQueryBuilder: same interface, different SQL dialect
  • Director class calls the same steps on any builder → different results
🧠 Exam Trick: Builder = step-by-step construction. Look for method chaining + final build(). Fixes "too many constructor params" problem. Different builders = different representations of same construction process.
Singleton
👑
"Ensure a class has only ONE instance and provide a global access point to it."

Why Only One?

  • Shared database connection — you don't want 10 separate connections
  • Logger — all logs go to same file/output
  • Config — all settings in one place, consistent app-wide

How It Works (The 3 Rules)

Rule 1: Constructor is PRIVATE (or protected)
Rule 2: Static field holds THE instance
Rule 3: Static getInstance() creates on first call, returns cached after

The Code Pattern (PHP Example)

class Singleton {
  private static $instances = [];
  
  protected function __construct() {} // can't use new!
  protected function __clone() {}   // can't clone!
  public function __wakeup() {
    throw new Exception("Cannot unserialize!");
  }
  
  public static function getInstance() {
    $subclass = static::class;
    if (!isset(self::$instances[$subclass])) {
      self::$instances[$subclass] = new static();
    }
    return self::$instances[$subclass];
  }
}

// Usage:
$l1 = Logger::getInstance();
$l2 = Logger::getInstance();
// $l1 === $l2 → TRUE! Same object!

From the Slides — Examples

  • Logger: writeLog(), static log() method, writes to file/stdout
  • Config: extends Singleton, hashmap[] for key-value settings, getValue() / setValue()
  • Client test: $l1 === $l2 → SAME instance confirmed
🧠 Exam Trick: Singleton = private constructor + static getInstance(). Three protections: no new, no clone, no unserialize. Check: if $l1 === $l2 → Singleton works!

Structural Patterns

About assembling objects into larger structures while keeping them flexible

Adapter
🔌
"Convert the interface of a class into another interface clients expect — make incompatible things work together."

The Problem

  • Stock market app reads XML data
  • New analytics library only works with JSON data
  • Can't modify the 3rd-party library!
  • Email app expects Notification interface, but Slack uses SlackApi

Components

«interface» Notification
+ send(title, message)
EmailNotification
implements Notification
(already compatible ✅)
SlackNotification
implements Notification
wraps SlackApi
(THE ADAPTER 🔌)
→ wraps
SlackApi
(3rd party)
login, sendMessage
Client: clientCode(Notification $n)
Works with ANY notification!

How SlackNotification Adapter Works

  • Implements the Notification interface (what client expects)
  • Constructor takes SlackApi instance (dependency injection)
  • send() → strips HTML tags, calls slackApi.login() then slackApi.sendMessage()
  • Client doesn't know it's talking to Slack!
WITHOUT Adapter
Client must know about SlackApi, EmailApi, SMSApi... every 3rd party!
WITH Adapter
Client only knows Notification interface. Adapters handle translation behind the scenes!
🧠 Exam Trick: Adapter = wrapper that translates one interface to another. Think: plug adapter for electricity. Key sign: "implements TargetInterface" + "has-a ThirdPartyClass". Not subclassing the 3rd party — wrapping it!
Decorator
☕
"Attach new behaviors to objects by placing them inside wrapper objects — without changing the original class."

The Problem

  • Basic Coffee costs 10. Add milk (+2), add cream (+5), add chocolate (+3)
  • Subclasses for each combo? → MilkCoffee, CreamCoffee, MilkCreamCoffee, MilkChocolateCoffee... 💥 explosion!
  • Decorators allow infinite combinations without new subclasses

Components & How They Connect

«interface» Coffee
+ getCost(): int
+ getDescription(): string
SimpleCoffee
cost=10
"Simple Coffee"
«abstract» CoffeeDecorator
implements Coffee
has: Coffee $decoratedCoffee
↓extends
MilkCoffee
+2 cost, ", milk"
CreamCoffee
+5 cost, ", cream"
ChocolateCoffee
+3 cost

Step by Step Example

  • SimpleCoffee → cost=10, desc="Simple Coffee"
  • new MilkCoffee(simpleCoffee) → cost = 10+2 = 12, desc = "Simple Coffee, milk"
  • new CreamCoffee(milkCoffee) → cost = 12+5 = 17, desc = "Simple Coffee, milk, cream"
  • Each decorator calls $this->decoratedCoffee->getCost() + own price

The KEY Pattern in Code

abstract class CoffeeDecorator implements Coffee {
  protected $decoratedCoffee; // wraps the coffee
  
  public function __construct(Coffee $coffee) {
    $this->decoratedCoffee = $coffee;
  }
}

class CreamCoffee extends CoffeeDecorator {
  const PRICE = 5;
  
  public function getCost(): int {
    // OWN price + wrapped coffee price
    return $this->decoratedCoffee->getCost() + self::PRICE;
  }
}
🧠 Exam Trick: Decorator = same interface, wraps itself. Both the original object AND the decorator implement the same interface. Key: "has-a" + "is-a" same interface simultaneously. Stacking = composing behaviors at runtime.

Behavioral Patterns

About communication and responsibility between objects

Observer
📡
"Define a subscription mechanism to notify multiple objects about events happening to the object they're watching."

The Problem

  • Customer wants to know when iPhone is in stock
  • Option 1: Customer visits store every day → Waste of time!
  • Option 2: Store emails ALL customers every day → Spam! Irrelevant!
  • Solution: Customer SUBSCRIBES, gets notified ONLY when relevant event happens

Components

Publisher (Subject / Observable)
EventManager
+ subscribe(eventType, listener)
+ unsubscribe(eventType, listener)
+ notify(eventType, data)
«interface» EventListener
+ update(filename)
←implements
LoggingListener
EmailAlertsListener
Editor (Observable)
has EventManager
openFile() → notify("open")
saveFile() → notify("save")

From the Slides — Editor Example

  • Editor = the subject/publisher (has EventManager)
  • LoggingListener = logs to file when event happens
  • EmailAlertsListener = sends email when event happens
  • editor.openFile() → events.notify("open", filename) → all subscribed listeners get update(filename)
  • Listeners don't know about each other! Loose coupling!

Key Pseudocode from Slides

// EventManager (the brain)
class EventManager:
  listeners = HashMap  // eventType → list of listeners
  
  subscribe(eventType, listener) → listeners.add(...)
  unsubscribe(eventType, listener) → listeners.remove(...)
  notify(eventType, data):
    foreach listener in listeners[eventType]:
      listener.update(data)

// Concrete Listener
class LoggingListener implements EventListener:
  update(filename):
    log.write("File opened: " + filename)

// Observable
class Editor:
  events = new EventManager()
  
  openFile(path):
    this.file = new File(path)
    events.notify("open", file.name)
🧠 Exam Trick: Observer = Publisher + Subscriber. Publisher doesn't care WHO is listening. Subscriber doesn't care about other subscribers. Key terms: subscribe(), notify(), update(). Real world: event systems, news feeds, notifications.
Strategy
⚽
"Define a family of algorithms, put each in its own class, and make them interchangeable at runtime."

The Problem — Football Match Example

  • Al Ahly uses Attack strategy (+2 bonus for winning)
  • Al Zamalek uses Defend strategy (+1 bonus for winning)
  • Before: Match class checks "if team is Ahly → attack, if Zamalek → defend" → violates OCP!
  • After: Each team has a strategy object injected — behavior changes without touching Match class

Components

«abstract» Strategy
- name
- scorePoint
+ play()
+ setScorePoint()
Attack extends Strategy
play() → "Attack!"
setScorePoint() → +2
Defend extends Strategy
play() → "Defend!"
setScorePoint() → +1
Team
- name, bonus, strategy
+ welcome() → strategy.name()
+ setBonus() → strategy.setScorePoint()
→ uses
Match
- team1, team2
+ play() → team1.play(), team2.play()
+ setScore(winner)
❌ BEFORE (Violates OCP)
if(team1 is Ahly) → attack
elif(team1 is Zamalek) → defend
→ add Bus Madrid? MODIFY MATCH!
✅ AFTER (Strategy Pattern)
team1.setStrategy(new Attack())
team2.setStrategy(new Defend())
match.play() → team1.play()
→ New team? Just new Strategy class!

The 3 Strategy Roles

  • Context (Team/Match): holds reference to Strategy, delegates behavior to it
  • Strategy Interface: common interface for all algorithms
  • Concrete Strategies (Attack, Defend): implement different versions of the algorithm
🧠 Exam Trick: Strategy = swap algorithms at runtime. Context has setStrategy(). All strategies share same interface. Fixes "if-else chains" that grow with every new case. Compare to State pattern: similar structure but Strategy is about algorithms, State about internal states.
⚡ Pattern Comparison

The exam LOVES to ask "which pattern should be used here?" — use this!

🏗️ Creational

  • Factory Method Need subclass to decide which object type
  • Builder Complex object, many optional params, step-by-step
  • Singleton Exactly ONE instance needed globally
  • Abstract Factory Families of related objects
  • Prototype Copy existing objects

🏛️ Structural

  • Adapter Incompatible interfaces need to work together
  • Decorator Add behaviors without subclassing, stackable
  • Bridge Separate abstraction from implementation
  • Composite Tree structures, treat individually or as group
  • Facade Simple interface to complex subsystem
  • Flyweight Share state to save memory

🧠 Behavioral

  • Observer Notify multiple objects about events
  • Strategy Swap algorithms at runtime
  • Chain of Resp. Pass request along handler chain
  • Command Turn request into standalone object
  • Iterator Traverse collection without exposing internals
  • Mediator Reduce direct dependencies between objects
  • Memento Save/restore object state (undo)
  • State Object changes behavior when state changes
  • Template Method Define algorithm skeleton, subclass fills steps

Factory Method

What problem does it solve?

👆 Click to reveal

Solves the problem of creating objects without specifying the exact class. The superclass defines the interface, but subclasses decide which concrete class to instantiate. Enables OCP: add new product types without modifying existing code.

Singleton

What are the 3 protection mechanisms?

👆 Click to reveal

1. Private/Protected constructor → can't use new

2. Protected __clone() → can't clone the instance

3. __wakeup() throws Exception → can't unserialize (which would create new instance)

Builder Pattern

What's the difference between Builder and Director?

👆 Click to reveal

Builder provides methods to construct each part of the object (buildWalls, buildDoors...) and returns the result via getResult().

Director is optional — it defines the order/sequence of build steps. The same director can work with different builders!

Adapter

How is it different from Decorator?

👆 Click to reveal

Adapter changes the interface — makes incompatible interface fit. The client sees a different interface than the adaptee has.

Decorator keeps the same interface — just adds behavior. The client sees the same interface, wrapped with extra functionality.

Observer

Name all 4 key components from the slides

👆 Click to reveal

1. Publisher/Subject (EventManager) — manages subscriber list
2. EventListener interface — update() method
3. Concrete Listeners (LoggingListener, EmailAlertsListener)
4. Observable (Editor) — has EventManager, calls notify() on events

Strategy

How does it fix OCP in the Football example?

👆 Click to reveal

Before (OCP violated): Match class had if/elif for every team type. Adding new team = modify Match class.

After (Strategy applied): Team receives a Strategy object (Attack/Defend). Match calls team.play() without knowing the strategy. Adding new strategy = new class only, Match unchanged!

Decorator

Why does CoffeeDecorator implement AND contain Coffee?

👆 Click to reveal

Implements Coffee (is-a): So the decorator is interchangeable with real coffee — client doesn't know the difference.

Contains Coffee (has-a): So it can delegate to the wrapped object and add to its behavior.

This double relationship is what makes stacking/chaining possible!

OCP Principle

State the principle + which patterns enforce it

👆 Click to reveal

Open/Closed Principle: Software entities should be open for extension (add new behavior by new classes) but closed for modification (don't change existing code).

Enforced by: Factory Method (add new products via new subclass), Strategy (add algorithms via new class), Decorator (add behavior via new wrapper)

🎯 Exam Killer Tips
01

Spot the Pattern by Problem

"Too many constructor params" → Builder. "Only one instance needed" → Singleton. "Incompatible interface" → Adapter. "Need to notify many objects" → Observer. "Swap algorithm" → Strategy.

02

OCP = Factory + Strategy

Any question about adding new types/behaviors WITHOUT modifying existing code → Factory Method or Strategy. If they ask "how do you add Bus transport?" → Create BusLogistics subclass, that's it!

03

getInstance() = ALWAYS Singleton

See getInstance() in code → Singleton, 100%. Also look for: private static $instances, protected constructor. The test: $a === $b (same reference = same instance).

04

Decorator vs Inheritance

Inheritance = static, compile-time. Decorator = dynamic, runtime. You can stack decorators. With inheritance you'd need MilkCreamChocolateCoffee class! Decorator wins for combinable behaviors.

05

Factory vs Abstract Factory

Factory Method = creates ONE product type (Transport). Abstract Factory = creates FAMILIES of related products (Button + Dialog + Checkbox all for Windows OR all for Mac).

06

Observer: Publisher doesn't know Subscribers

KEY exam point: Publisher only knows the EventListener interface. It doesn't know if it's logging, emailing, or SMSing. Subscribers are added/removed dynamically. This is loose coupling!

07

Builder = Fluent Interface

Method chaining (return $this) is the signature. Always ends with build() or getResult(). SQL Builder: .select().where().limit().getSQL(). Car Builder: .setYear().setColor().build().

08

Adapter: Has-a + Implements

SlackNotification IMPLEMENTS Notification (target interface) and HAS-A SlackApi (adaptee). The constructor takes SlackApi by dependency injection. strip_tags() in send() = adapt the data format too!

📐 Key Principles Recap

Open/Closed Principle (OCP)

Open for extension, closed for modification. Add new behavior via new classes, not by editing existing ones. Violated when you see growing if/elif/instanceof chains.

Single Responsibility Principle

Each class does ONE thing. SlackApi handles Slack API calls. SlackNotification handles adapting to Notification interface. They don't mix!

Dependency Injection

Instead of creating dependencies inside a class, pass them in (usually constructor). Seen in: Adapter (SlackApi injected), Observer (EventManager composed in Editor).

Loose Coupling

Objects depend on interfaces, not concrete classes. Client uses Notification interface → works with Email, Slack, SMS, anything that implements it. Change one, others unchanged.
